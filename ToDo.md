# Todo / Known Issues

## Bug: Stale devices produce infinite repeated log messages

**File:** `main.go` — `checkForStaleMetrics()`

`checkForStaleMetrics` deletes Prometheus gauge labels when a device goes stale but never removes the device from `lastUpdateTime`. On every subsequent `RefreshInterval` tick (default every 30s), the device is still found in `lastUpdateTime`, still considered stale, and the "Metrics for device X reset due to inactivity" log line fires again — indefinitely. `DeleteLabelValues` is also called repeatedly (harmless but wasteful).

**Fix:** After deleting the gauge labels for a stale device, also delete its entry from `lastUpdateTime`.

## Bug: Hot-reload writes to the global Viper instance concurrently

**File:** `config.go` — `watchConfigFile()` → `initConfig()`

`watchConfigFile` fires `initConfig()` from a debounce timer goroutine. `initConfig()` calls `viper.SetDefault(...)`, `viper.ReadInConfig()`, `viper.AutomaticEnv()`, `viper.BindEnv()`, etc. on the **global Viper instance**. Viper documents concurrent reads as safe but concurrent writes are not. Every hot-reload performs many write operations on global state, which can race against concurrent reads elsewhere.

**Fix:** Replace all package-level `viper.*` calls in `initConfig()` with a locally created `v := viper.New()` instance, so each call is fully isolated.

## Bug: `parseGoveeData` acquires the mutex three separate times

**File:** `main.go` — `parseGoveeData()`

For a single BLE advertisement, the function acquires and releases `mutex` three times:

1. Read/write `deviceLastLoggedVals`
2. Iterate `knownGovees` to compute log column padding width
3. Write `lastUpdateTime` and call `setDeviceStatusLocked`

Between locks, a concurrent hot-reload can modify `knownGovees`, making the log output inconsistent and the status update potentially stale relative to what was logged. The three operations should ideally be performed under a single lock.

## Test: `TestConfigInitialization` is sensitive to `config.yaml` content

**File:** `config_test.go` — `TestConfigInitialization()`

The first sub-test asserts that `initConfig()` returns default values. However, `config.yaml` exists in the repo and Viper loads it automatically, overriding defaults. The test passes currently only because the repo's `config.yaml` happens to match the expected default values. Any change to `config.yaml` (e.g., tuning scan intervals) will silently cause the "default values" assertions to test config-file values instead of actual defaults.

**Fix:** The test should either temporarily rename/remove `config.yaml`, point Viper at a temp directory with no config file, or use a separate `initConfigFromFile(path)` function that can be given an empty path for testing.

## Maintenance: Build artifacts committed alongside sources

**Files:** `static/app.js`, `static/styles.css`

These are generated by `build-js.js` and `build-css.js` from sources in `static/js/` and `static/css/`. Both generated files are committed to git alongside their sources with no CI step to verify they are in sync. Editing a source file without running `make build-css build-js` will leave stale assets served to users running the app outside Docker.

**Options:**

- Add a CI check that runs the build and fails if the output differs from what is committed.
- Or add `static/app.js` and `static/styles.css` to `.gitignore` and have the Dockerfile/entrypoint run the build step.

## Design Gap: Bluetooth timing and stale-metrics config not hot-reloaded

**File:** `main.go` — BLE scanner goroutine and stale metrics checker goroutine

Both goroutines capture the initial `*Config` at startup and never receive updates:

- `startBLEScanner` uses `config.Bluetooth.ScanInterval` and `config.Bluetooth.ScanDuration` from the original config.
- The stale metrics ticker uses `config.Metrics.RefreshInterval` and `checkForStaleMetrics` uses `config.Metrics.StaleThreshold` from the original config.

Changes to these values in `config.yaml` are silently ignored until a restart. No warning is logged to inform the operator of this limitation, unlike the OpenMeteo poller which does support dynamic interval updates.

**Fix (minimal):** Log a warning when these fields change during a hot-reload indicating a restart is required. A fuller fix would share the config via a protected pointer (like `currentConfig`/`currentConfigMu`) and have the goroutines re-read it on each iteration.
